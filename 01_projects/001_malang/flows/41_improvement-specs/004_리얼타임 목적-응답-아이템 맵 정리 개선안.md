# 리얼타임 목적-응답-아이템 맵 정리 개선안

## 왜 이게 중요한가
- 현재 이벤트 분기 버그의 상당수가 "맵 정리 타이밍" 문제에서 시작된다.
- 특히 `response.done`, `response.canceled`, `conversation.item.deleted` 순서가 바뀌면 stale mapping이 남기 쉽다.

## 현재 맵(관찰 대상)
- `responsePurposeById`
- `activeResponseIdByPurpose`
- `latestAssistantItemIdByPurpose`
- `assistantItemCreatedAt`

## 문제 패턴
- done 전에 canceled가 오거나, deleted가 먼저 오면 역정리 실패 가능
- active 포인터가 새 응답으로 교체됐는데 이전 응답 cleanup이 늦게 들어오는 경우
- purpose별 단일 active 가정이 깨지는 순간(재시도/중복 응답) 처리 취약

## 개선 원칙
- 이벤트 순서 역전에 안전한 "멱등 cleanup"으로 작성
- 맵 정리는 한 곳(`cleanupRealtimeMaps`)으로 집중
- 맵 수명주기를 명시: `created -> active -> completed/canceled -> gc`
- idempotency 키/중복 실행 방지 전략과 함께 적용

## 제안 API(예시)
```ts
interface RealtimeMapState {
  responsePurposeById: Map<string, string>;
  activeResponseIdByPurpose: Map<string, string>;
  latestAssistantItemIdByPurpose: Map<string, string>;
  assistantItemCreatedAt: Map<string, number>;
}

function registerResponseCreated(state: RealtimeMapState, responseId: string, purpose: string): void;
function registerAssistantItem(state: RealtimeMapState, responseId: string, itemId: string): void;
function cleanupByResponseId(state: RealtimeMapState, responseId: string): void;
function cleanupByItemId(state: RealtimeMapState, itemId: string): void;
function gcStaleMappings(state: RealtimeMapState, now: number): number;
```

## 정리 규칙(핵심)
- `cleanupByResponseId`는 여러 번 호출돼도 결과가 같아야 한다(idempotent)
- active 포인터 삭제 시 "현재 active와 동일한 경우만" 제거
- `item.deleted`는 purpose 역추적 후 관련 item 맵만 제거
- 주기적 GC로 오래된 orphan item 정리

## 적용 순서
- 1) 기존 `handleRealtimeMessage` 내부 cleanup 코드를 함수화
- 2) response/item cleanup 호출 지점을 공통 함수로 통일
- 3) 테스트 시나리오 추가(done->cancel 역순, delete 선행, 중복 cancel)

- active response 포인터 mismatch 로그가 유의미하게 감소
- 수동 테스트 중 stale cancel/delete 오작동 재현률 감소
