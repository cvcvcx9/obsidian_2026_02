# 왜 이 개선이 필요한가

## 한 줄 결론
- 이번 개선은 "코드를 예쁘게" 만들기 위한 것이 아니라, **실제 런타임에서 발생하는 중복 실행/순서 역전/맵 꼬임으로 인한 오동작을 줄이기 위한 안정화 작업**이다.

## 개선 전 실제 문제 패턴
- Realtime 특성상 같은 의미 이벤트가 중복 수신될 수 있어 `sendFsmEvent`가 중복 호출될 수 있었다.
- `response.done`, `response.canceled`, `conversation.item.deleted` 순서가 바뀌면 response/item 맵 정리 타이밍이 흔들렸다.
- 이벤트 분기/정리 로직이 여러 위치에 퍼져 있어, 한 군데 수정 시 다른 경로 회귀 가능성이 높았다.
- 결과적으로 아래와 같은 증상이 생길 수 있었다.
  - 같은 FSM 전이가 2번 발생
  - cancel/delete가 stale 대상에 적용
  - active response 포인터가 남거나 잘못 지워짐

## 이번 단계에서 꼭 한 이유
- **Idempotency 적용**: 같은 이벤트/액션/FSM 호출의 중복 실행을 먼저 차단해 장애 전파를 막음.
- **맵 정리 공통화**: response/item 정리 로직을 함수로 모아 역순 이벤트에도 멱등(no-op 안전)하게 처리.
- **운영 가시성 확보**: duplicate skip 로그를 통해 "중복이 실제로 차단되는지" 확인 가능.

## 기대되는 효과
- 중복 클릭/중복 수신 상황에서도 side effect가 1회로 수렴.
- 역순 이벤트에서 최종 상태 일관성 유지.
- 원인 추적 속도 개선(이벤트 수신 문제 vs 맵 정리 문제 구분 쉬움).
- 다음 단계(책임분리, 중복코드 제거) 진행 리스크 감소.

## 지금 기준으로 남은 이유(계속 해야 하는 이유)
- 아직 stale mapping GC 규칙이 남아 있어 장시간 세션/예외 시나리오 안정화를 위해 필요.
- 서버 레벨 idempotency 스펙 연동이 확정되면 프론트/서버 양쪽에서 중복 방어가 완성됨.
