# 전사 가드 로직 공통화

## 왜 필요한가
- `conversation.item.input_audio_transcription.completed` 처리 블록에 가드 조건이 길게 섞여 있어 읽기/수정 비용이 높았다.
- 같은 종류의 판단(저신호, 중복, speech_started 누락 fallback)이 분기 내부에 흩어져 회귀 위험이 있었다.

## 어떤 작업을 했는가
- 가드 판단을 `decideTranscriptGuard(...)`로 분리했다.
- 반환값(`TranscriptGuardDecision`)으로 아래를 명시한다.
  - 수용 여부(`accepted`)
  - 거부 사유(`reason`)
  - 누락 fallback 사용 여부(`usedMissingSpeechFallback`)
  - `speechStartedAt` 클리어 필요 여부(`shouldClearSpeechStarted`)
- 기존 이벤트 처리에서는
  - 카운터 업데이트
  - 가드 함수 호출
  - 로그/조기 return
  - 성공 시 후속 처리
  순서로 단순화했다.

## 기대 효과
- 가드 정책 변경 시 수정 지점이 한 곳으로 줄어든다.
- 가드 사유별 로그 흐름이 명확해져 디버깅이 쉬워진다.
- 다음 단계(action 유틸화/로그 키 표준화) 적용 시 충돌을 줄인다.
