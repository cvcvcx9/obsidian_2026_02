# 맵 정리 공통화

## 목적
- Realtime 응답/아이템 맵 정리 로직을 한 곳으로 모아 중복 구현과 순서 역전 버그를 줄인다.

## 왜 분리했는가
- 기존에는 `response.done`, `response.canceled`, `conversation.item.deleted`에서 비슷한 정리 코드를 각각 가지고 있었다.
- 이 구조는 "한 군데만 수정"되거나 "정리 규칙이 조금씩 달라지는" 회귀를 만들기 쉬웠다.

## 공통화 대상 맵
- `responsePurposeById`
- `activeResponseIdByPurpose`
- `latestAssistantItemIdByPurpose`
- `assistantItemCreatedAt`

## 도입한 공통 함수
- `registerResponsePurposeMapping(responseId, purpose, eventId?)`
  - response 생성 시 purpose/active 포인터 등록
  - 필요 시 `requestPurposeByEventId` 정리
- `registerAssistantItemMapping(itemId, responseId?)`
  - assistant item 생성 시 purpose 기반 item 매핑 등록
- `cleanupResponseMapping(responseId, reason)`
  - response 종료/취소/실패 시 목적 맵 정리
  - active 포인터는 "현재 active와 일치할 때만" 삭제
- `cleanupAssistantItemMapping(itemId, reason)`
  - item 삭제/정리 시 item 관련 맵 정리
  - 여러 번 호출돼도 안전한 no-op 처리

## 적용 지점
- `response.created` -> `registerResponsePurposeMapping`
- `conversation.item.created(role=assistant)` -> `registerAssistantItemMapping`
- `response.done` -> `cleanupResponseMapping`
- `response.canceled/cancelled/failed` -> `cleanupResponseMapping`
- `conversation.item.deleted` -> `cleanupAssistantItemMapping`
- `purgeRecentAssistantItemByPurpose` 성공 시 -> `cleanupAssistantItemMapping`

## 기대 효과
- 정리 규칙이 한 곳에 모여 유지보수 난이도 감소
- 이벤트 순서 역전(`done -> cancel`, `cancel -> done`)에도 멱등 정리 가능
- stale 포인터/맵 누수 디버깅이 쉬워짐

## 남은 작업
- stale mapping GC 규칙 추가
- 역순/중복 이벤트 케이스 검증 시나리오 문서화
